# ПРОМТ: bitaps VPN — приложение под Linux, Windows, Android

Это полное техническое задание для создания кроссплатформенного клиента **bitaps VPN** под **Linux, Windows и Android**. Приложение должно быть визуально и функционально единым с уже существующими нативными клиентами (iOS/macOS, SwiftUI) и с сайтом `bitaps-vpn.surge.sh`. **Все платформы делят ОДИН бэкенд и ОДНУ аккаунт-систему** — новый клиент ничего на сервере не создаёт, только привязывается к существующему.

Скопируй этот файл целиком как контекст для исполнителя (человека или агента) на другом аккаунте.

## 0. ЦЕЛЬ

Сделать нативно-ощущаемое VPN-приложение под Linux / Windows / Android, которое:
- выглядит как iOS/macOS-версия и сайт (тёмно-космическая эстетика, оранжевый акцент, шестерёнка-логотип с буквой **B**);

## 1. ТЕХНОЛОГИЧЕСКИЙ СТЕК (рекомендация)

**Один кодовый базис на все три платформы → Flutter (Dart).** Flutter нативно собирает под Android, Windows и Linux. Это даёт единый UI и логику.

- **UI:** Flutter + кастомная дизайн-система (см. §5). НЕ Material-дефолт — рисуем свои компоненты под бренд.
- **VPN-движок:** **sing-box** (тот же, что планируется в iOS/macOS-клиенте). Подключать через `libbox` (gomobile/gomobile-bind для Android; C-shared `.so`/`.dll` через FFI для Linux/Windows). Готовый проверенный референс — **Hiddify** (open-source, Flutter + sing-box) и **sing-box** официальные обёртки. Можно форкать подход Hiddify, но UI/бренд — наш.
- **Backend SDK:** `supabase_flutter` (Dart-клиент Supabase) для auth/REST.
- **Хранение настроек:** `shared_preferences` (аналог UserDefaults).

> Альтернатива, если Flutter не подходит: Tauri (Rust+web) для десктопа + Kotlin для Android — но это два кодовых базиса. **Рекомендуется Flutter** ради единства.

Платформенные особенности VPN:
- **Android:** `VpnService` API (требует разрешения пользователя, иконка-ключ в статус-баре). sing-box через `libbox.aar`.
- **Windows:** TUN через WinTUN (sing-box умеет), запуск с правами/службой; иконка в трее.
- **Linux:** TUN-устройство (sing-box), нужен `CAP_NET_ADMIN` / polkit; иконка в трее (AppIndicator).

## 3. АУТЕНТИФИКАЦИЯ (через Telegram)

Новый клиент логинит юзера через Telegram и получает Supabase-сессию:

1. Показать **Telegram Login** (widget в WebView, либо deep-link флоу через бота). Получить подписанный payload (`id, first_name, …, auth_date, hash`).
2. `POST /functions/v1/telegram-auth` с этим объектом → ответ `{ email, token_hash }`.
   (Функция валидирует HMAC-SHA256 по bot-token, создаёт/находит юзера `tg<ID>@telegram.bitaps.local`, кладёт `telegram_id` в `user_metadata`.)
3. Завершить вход: `supabase.auth.verifyOtp({ token_hash, type: 'magiclink' })` → получить сессию (JWT с `user_metadata.telegram_id`). Хранить/рефрешить как обычную Supabase-сессию.

> **Email/Google вход существует только на сайте и НЕ привязан к подписке** (нет telegram_id). Нативные клиенты ОБЯЗАНЫ логиниться через Telegram, иначе не увидят подписку/ключ.
> Дополнительно есть «Демо»-режим (локальные мок-данные) для первого знакомства без входа.

## 4. ОБЩАЯ API-ПОВЕРХНОСТЬ (что дёргает клиент)

Base = `https://bjkozsukvifkxriojxrz.supabase.co`, header `apikey: sb_publishable_X2CJWgjqeZtbNelAri9ofw_trbfWF9Z`.

**4.1 Войти** — см. §3 (`telegram-auth` → `verifyOtp`).

**4.2 Подписка + лимит устройств + ключ** (с сессией):
```
GET /rest/v1/subscriptions?select=plan,expires_at,vpn_key,device_limit
   Authorization: Bearer <session JWT>
   apikey: <anon>
```
RLS вернёт только свою строку. `active = expires_at > now()`.

**4.3 VLESS-конфиг** — это поле `vpn_key` из 4.2 (строка `vless://…`). Один ключ на все устройства аккаунта. Импортировать в sing-box. (См. §12 — сейчас демо.)

**4.4 Лидерборд рыбалки:**
```
GET  /rest/v1/game_scores?select=name,score&order=score.desc&limit=5   (header apikey)
POST /rest/v1/game_scores   (apikey, Content-Type: json, Prefer: return=minimal)  body {name, score}
```
Имя чистить до `[A-Za-zА-Яа-яЁё0-9]`, ≤24.

**4.5 Оплата** — самый простой путь: deep-link в `@bitaps_vpn_auth_bot` (оплата Stars/Crypto/СБП в боте), затем перечитать 4.2. Веб-оплату для нативных клиентов НЕ использовать (она шлёт ключ на почту, а не в аккаунт).

**4.6 Заказ B-box / поддержка** (опц.): `POST /functions/v1/box-order` `{initData, action:"order"|"support", …}` (нужна Telegram initData). Вне Telegram — проще форма на почту `bitapssupport@gmail.com`.

## 12. ЧТО НА БЭКЕНДЕ ЕЩЁ НЕ ГОТОВО (важно указать)

🔴 **`vpn_key` сейчас — ДЕМО-заглушка.** Реального VLESS+Reality сервера за ним нет: host/`pbk`(public key)/`sni` — плейсхолдеры (`vpn.bitaps.app:443`, `pbk=DEMO…`). Поэтому:
- Приложение можно полностью собрать и довести UI/логику на этом демо-ключе (туннель поднимется, но трафик в реальный сервер не пойдёт).
- Для боевого VPN нужно (отдельная задача, общая для всех платформ): **поднять VLESS+Reality сервер(ы)** и выдавать настоящий per-user конфиг (генерацию вынести в Edge-функцию/панель), вернуть реальные `pbk/sni/host`. Это единственная по-настоящему недостающая часть бэкенда.
- Реального учёта подключённых устройств нет — только число `device_limit` (клиент сам не превышает).
